# 분탕글 판별 2차 MVP 구현 작업 목록

## 1. 목적

이 문서는 [2차 MVP 설계서](/C:/project/dactest/docs/troll-post-detection-mvp-2.md:1)를 실제 구현 가능한 작업 단위로 쪼갠 목록이다.

목표는 아래 세 가지다.

1. 작업을 바로 순서대로 구현할 수 있을 것
2. 각 작업의 완료 기준이 명확할 것
3. 3차 학습 단계로 넘어가기 위한 데이터 축적 준비를 끝낼 것

## 2. 작업 단위 원칙

- 한 작업은 가능한 한 하나의 책임만 가진다
- 서버/프론트/DB 작업을 분리한다
- 각 작업은 확인 가능한 완료 기준을 가진다
- 위험한 작업은 선행 검증 단계를 둔다

## 3. 전체 순서

1. 의존성 및 디렉터리 준비
2. SQLite 초기화 계층 추가
3. 게시글 스냅샷/예측 저장 계층 추가
4. 라벨 저장 계층 추가
5. 상세 API에 저장/조회 연동
6. 라벨 API 추가
7. 프론트 상세 라벨 UI 추가
8. 목록에 라벨 상태 반영
9. 통계 API와 선택적 표시 추가
10. 수동 검증 및 데이터 확인

## 4. 구현 작업 목록

## 4.1 작업 A: 의존성 추가

### 목표

SQLite 사용을 위한 라이브러리를 추가한다.

### 작업 내용

- `package.json`에 `better-sqlite3` 추가
- 필요 시 `data/` 디렉터리 생성

### 완료 기준

- `require("better-sqlite3")`가 서버에서 정상 로드된다
- `data/` 경로가 준비된다

### 산출물

- [package.json](/C:/project/dactest/package.json:1)

## 4.2 작업 B: DB 초기화 모듈 추가

### 목표

앱 시작 시 SQLite DB와 테이블이 준비되도록 한다.

### 작업 내용

- `lib/db.js` 생성
- DB 파일 경로 결정
- 테이블 생성 SQL 작성
- 앱 시작 시 초기화 수행

### 포함 테이블

- `posts_snapshot`
- `post_prediction`
- `post_label`
- `label_event`

### 완료 기준

- 서버 시작 시 `data/app.db`가 생성된다
- 모든 테이블이 자동 생성된다
- 재시작해도 오류 없이 유지된다

### 산출물

- `lib/db.js`

## 4.3 작업 C: 게시글 스냅샷 저장 계층 추가

### 목표

상세 조회 시점의 글 원문 데이터를 저장할 수 있게 한다.

### 작업 내용

- `lib/post-store.js` 생성
- `savePostSnapshot(post)` 구현
- `getPostSnapshot(postNo)` 구현
- `upsertPrediction(postNo, risk, version)` 구현

### 저장 항목

- 글 번호
- 제목
- 카테고리
- 작성자
- 날짜
- 본문 HTML
- 본문 텍스트
- 조회/추천/댓글 수
- 수집 시각

### 완료 기준

- 서버 코드에서 상세 글 1건 저장 호출 가능
- 같은 글 재저장 시 중복 insert가 아니라 upsert로 동작

### 산출물

- `lib/post-store.js`

## 4.4 작업 D: 라벨 저장 계층 추가

### 목표

사용자 라벨과 라벨 변경 이력을 저장할 수 있게 한다.

### 작업 내용

- `lib/label-service.js` 생성
- `setLabel(postNo, label)` 구현
- `clearLabel(postNo)` 구현
- `getLabel(postNo)` 구현
- `getLabelStats()` 구현

### 완료 기준

- 라벨 저장, 수정, 취소가 모두 가능하다
- `label_event`에 변경 이력이 남는다
- 현재 최종 라벨과 이력이 분리되어 관리된다

### 산출물

- `lib/label-service.js`

## 4.5 작업 E: 서버 시작 시 DB 초기화 연결

### 목표

앱이 켜질 때 DB 계층이 자동 준비되도록 한다.

### 작업 내용

- [server.js](/C:/project/dactest/server.js:1)에 `db.js` 연결
- 초기화 실패 시 명확한 오류 로그 출력

### 완료 기준

- 서버 시작 시 DB 준비가 자동 수행된다
- 초기화 실패 시 원인 파악이 가능하다

## 4.6 작업 F: 상세 API에 스냅샷/예측 저장 연동

### 목표

상세 글을 열면 자동으로 학습 준비 데이터가 쌓이게 한다.

### 작업 내용

- `/api/posts/:no`에서 상세 글 파싱 후 저장 호출
- `posts_snapshot` upsert
- `post_prediction` upsert
- 현재 `userLabel` 조회 후 응답에 포함

### 완료 기준

- 상세 API 호출 후 DB에 스냅샷과 예측 결과가 저장된다
- 응답 JSON에 `userLabel`이 포함된다

### 산출물

- [server.js](/C:/project/dactest/server.js:1)

## 4.7 작업 G: 라벨 저장 API 추가

### 목표

프론트가 라벨을 저장/수정/취소할 수 있도록 API를 연다.

### 작업 내용

- `POST /api/posts/:no/label`
- `DELETE /api/posts/:no/label`
- 입력값 검증
- `label` 값 검증: `troll | normal`

### 예외 처리

- 잘못된 라벨 값이면 `400`
- 저장 실패 시 `500`
- 대상 글 스냅샷이 없으면 먼저 확보하거나 오류 반환

### 완료 기준

- API 테스트로 분탕/정상/취소가 모두 동작한다
- 응답에 저장 결과와 시간 정보가 포함된다

## 4.8 작업 H: 라벨 통계 API 추가

### 목표

누적 라벨 데이터의 상태를 간단히 확인할 수 있게 한다.

### 작업 내용

- `GET /api/labels/stats`

### 응답 항목

- 전체 라벨 수
- 분탕 라벨 수
- 정상 라벨 수
- 마지막 집계 시각

### 완료 기준

- 라벨 저장 후 통계가 즉시 반영된다

## 4.9 작업 I: 상세 화면 라벨 버튼 UI 추가

사용자가 상세 화면에서 직접 라벨링할 수 있게 한다.

## 4.10 작업 J: 상세 화면 라벨 상태 표시

### 목표

사용자가 현재 라벨 상태를 쉽게 알 수 있게 한다.

### 작업 내용

## 4.11 작업 K: 목록 화면 라벨 상태 반영

### 목표

상세에서 라벨한 글이 목록에서도 구분되게 한다.

### 작업 내용

- 목록 아이템에 작은 라벨 상태 표시 추가
- 자동 판별 배지보다 낮은 우선순위로 시각 처리

### 권장 문구

### 완료 기준

## 4.12 작업 L: 프론트 상태 관리 정리

### 목표

라벨 동작 중 화면 흔들림 없이 상태가 유지되게 한다.

### 작업 내용

- `currentArticleLabel`
- `isSavingLabel`
- `labelStats` 추가
- 상세 갱신/목록 갱신 간 상태 일관성 확보

### 완료 기준

- 연속 클릭 시 중복 저장이 방지된다
- 저장 중 UI가 일관되다

## 4.13 작업 M: 저장 전 스냅샷 보장 처리

### 목표

라벨만 있고 원문 스냅샷이 없는 상태를 막는다.

### 작업 내용

- 라벨 저장 전 `posts_snapshot` 존재 여부 확인
- 없으면 상세 조회 저장 루틴을 먼저 수행하거나 저장 거부

### 완료 기준

- DB에서 라벨 데이터만 단독으로 남지 않는다

## 4.14 작업 N: 예측 버전 저장

향후 규칙 튜닝 효과를 추적할 수 있게 한다.

### 작업 내용

- `post_prediction` 저장 시 함께 기록

### 권장 값

### 완료 기준

## 8. 완료 기준

2차 MVP 구현 완료는 아래 조건을 만족할 때로 본다.

1. 사용자가 상세 화면에서 `분탕글/정상글/취소`를 할 수 있다
2. 서버가 글 스냅샷, 예측 결과, 최종 라벨, 라벨 이력을 DB에 저장한다
3. 새로고침 후에도 라벨 상태가 유지된다
4. 최소한의 통계 조회가 가능하다
5. 이후 3차 학습에 사용할 데이터가 누락 없이 쌓인다
