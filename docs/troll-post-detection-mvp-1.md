# 분탕글 판별 1차 MVP 설계서

## 1. 문서 목적

이 문서는 분탕글 판별 기능의 1차 MVP를 실제 개발 가능한 수준으로 구체화한다.

대상 프로젝트는 현재 [server.js](/C:/project/dactest/server.js:1) 기반의 단일 Node.js 서버와 [public/app.js](/C:/project/dactest/public/app.js:1) 기반의 단순 프론트엔드 구조다.

이번 MVP는 다음 목표에 집중한다.

- 새 글 목록에서 분탕 가능성 점수를 보여준다
- 글 상세 화면에서 점수와 판별 근거를 보여준다
- 규칙 기반 분류기를 도입해 운영자가 기준을 빠르게 조정할 수 있게 한다

이번 MVP에서는 구현 복잡도를 낮추기 위해 학습 기능은 제외한다.

## 2. MVP 범위

### 포함

- 규칙 기반 분탕글 점수 계산
- 목록 API 응답에 분탕 점수 포함
- 상세 API 응답에 분탕 점수와 근거 포함
- 목록 화면 배지 표시
- 상세 화면에 점수, 등급, 판별 근거 표시
- 점수 기준 문구 정리

### 제외

- 사용자 라벨 저장
- DB 도입
- 자동 재학습
- 모델 버전 관리
- 관리자 전용 규칙 편집 UI

### 확장 옵션

사용자 라벨 버튼은 별도 저장소가 필요하므로 엄밀히 말하면 2차에 가깝다. 다만 원하면 `localStorage` 기반 임시 기능으로 1.5차 수준의 빠른 검증은 가능하다.

## 3. MVP 목표 정의

MVP의 목적은 “정확한 AI 분류기 완성”이 아니라 아래 두 가지를 빠르게 검증하는 것이다.

1. 어떤 규칙 조합이 실제로 유의미한 신호를 주는지 확인
2. 운영 화면에서 분탕 가능성 점수를 보여주는 방식이 실용적인지 확인

따라서 1차 MVP의 핵심은 `설명 가능한 점수화`다.

## 4. 사용자 경험 목표

### 목록 화면

각 글 항목에 다음 정보를 추가한다.

- `분탕 낮음`
- `분탕 주의 38%`
- `분탕 높음 74%`

표시 원칙:

- 30% 미만은 너무 시각적으로 강하게 강조하지 않는다
- 60% 이상부터 눈에 띄는 색상 사용
- 80% 이상은 강한 경고색 사용

### 상세 화면

글 상단 메타 영역 아래에 카드 형태로 표시한다.

- 분탕 가능성 74%
- 등급: 높음
- 판별 근거:
  - 정치 키워드 2건
  - 강한 비하 표현 1건
  - 특정 자격증 폄하 패턴 1건

설명 문구:

- `이 수치는 자동 판별 점수이며 실제 판단과 다를 수 있습니다.`

- 등급 산정
- 응답용 요약 생성

## 6.2 분석 입력 구조

```js
{
  no: "131551",
  category: "일반",
  title: "정처기 왜 따냐",
  contentText: "쓸모없는데 왜 따냐 ...",
  commentsText: [],
  replyCount: 3
}
```

초기 MVP에서는 댓글 내용은 점수에 반영하지 않는 것을 권장한다.

이유:

- 목록에서는 댓글 본문이 없음
- 상세에서만 댓글이 있어 목록/상세 점수 일관성이 깨질 수 있음
- 우선 제목과 본문 기준으로 안정적인 동작을 만드는 것이 더 중요함

즉, 1차 MVP는 `제목 + 본문 중심`으로 간다.

## 6.4 점수 계산 방식

최종 점수는 단순 합산 후 0~100 사이로 제한한다.

```txt
score = min(100, max(0, sum(weights)))
```

1차 MVP에서는 확률 보정(calibration) 없이 정수 점수로만 사용한다.

## 6.5 등급 기준

- `low`: 0~29
- `medium`: 30~59
- `high`: 60~79
- `critical`: 80~100

UI 노출 문구:

- `low` -> `낮음`
- `medium` -> `주의`
- `high` -> `높음`
- `critical` -> `매우 높음`

## 7.2 규칙 카테고리

### 광고 패턴

대표 신호:

- URL 또는 도메인 포함
- 오픈채팅/텔레그램/디스코드 언급
- `문의`, `상담`, `홍보`, `부업`, `수익`, `지원금` 류 표현
- 연락 유도 패턴

예시 가중치:

- 외부 유도 키워드 1개 이상: +15
- 광고 키워드 2개 이상: +20
- URL 포함 + 광고 키워드 동시 등장: +20 추가

### 정치글 패턴

대표 신호:

- 정치인명
- 정당명
- 선거, 탄핵, 좌파/우파 등 정치 프레임 표현

예시 가중치:

- 정치 키워드 1개: +12
- 정치 키워드 2개 이상: +25
- 강한 정치 비하 표현 동반: +15 추가

### 정처기 폄하형 분탕 패턴

핵심은 “정처기 무용론 자체”가 아니라 `공격적/도발적 톤`과의 조합이다.

대표 신호:

- `정처기`, `정보처리기사`, `기사` 등 대상 키워드
- `쓸모없`, `왜 따`, `필요없`, `버려라`, `시간낭비` 등 부정 키워드
- 비하/조롱 단어 동반

예시 가중치:

- 정처기 키워드 + 부정 키워드 동시 등장: +18
- 여기에 공격적 표현 동반: +20 추가

### 매우 심한 비하/조롱 패턴

대표 신호:

- 강한 욕설 사전
- 조롱성 반복 표현
- 특정 대상을 지칭하며 공격하는 구조

예시 가중치:

- 강한 욕설 1회: +15
- 강한 욕설 2회 이상: +25
- 욕설 + 특정 대상 지칭 조합: +15 추가

## 7.3 완화 규칙

오탐을 줄이기 위해 감점 규칙도 필요하다.

예시:

- `후기`, `비교`, `분석`, `공부`, `합격`, `질문` 같은 정보성 문구 포함: -5~-15
- 제목은 자극적이어도 본문이 차분한 설명형이면 일부 감점
- 정치 키워드 1건만 있고 본문이 짧지 않으며 정보형이면 감점

초기 MVP에서는 복잡하게 하지 말고 2~3개 감점 규칙만 둔다.

## 7.4 1차 사전 관리 방식

코드 내부 상수로 시작한다.

예시:

```js
const KEYWORDS = {
  ad: ["오픈채팅", "텔레", "디스코드", "홍보", "문의", "상담"],
  politics: ["대통령", "국힘", "민주당", "좌파", "우파", "탄핵"],
  certTarget: ["정처기", "정보처리기사", "기사"],
  certNegative: ["쓸모없", "필요없", "시간낭비", "왜 따"],
  abuseStrong: ["병신", "개돼지", "쓰레기"]

```

1차에서는 파일 외부 설정까지는 가지 않고, 코드 상수로 시작하는 것이 개발 속도상 유리하다.

## 11. 성능 고려사항

목록 API는 주기적으로 호출되므로 계산 비용을 가볍게 유지해야 한다.

권장 방식:

- 목록에서는 제목 + 카테고리만으로 1차 점수 계산
- 상세에서는 제목 + 본문으로 재계산

이 방식의 장점:

- 목록 응답 속도 유지
- 상세에서 더 정확한 설명 제공 가능

단점:

- 목록 점수와 상세 점수가 일부 다를 수 있음

이를 줄이기 위해 권장하는 UI 문구:

- 목록: `분탕 주의`
- 상세: `분탕 가능성 74%`

즉, 목록은 경향성 표시, 상세는 설명형 표시로 역할을 나눈다.

## 12. 오탐 방지 기준

아래 글은 바로 고위험으로 올리지 않도록 주의한다.

- 시험 난이도 토론
- 자격증 효용성 토론
- 합격/불합격 경험담
- 채용시장 평가 글
- 뉴스 공유 글

1차 MVP 기준:

- 특정 키워드 하나만으로 60점 이상 주지 않는다
- “정처기 + 부정 표현”만으로는 `medium`에 머물고, 강한 조롱이 동반될 때 `high`로 올린다
- 정치 키워드 1건만으로는 `low~medium`에 머물도록 한다

## 13. 테스트 시나리오

### 정상 글

- `정처기 실기 공부법 공유`
- 기대 결과: `low`

### 정보성 비판 글

- `실무에서 정처기 효용이 낮은 이유`
- 기대 결과: `low` 또는 `medium`

### 분탕성 무용론 글

- `정처기 왜 따냐 진짜 쓸모없는 자격증인데`
- 기대 결과: `high`

### 광고글

- `고수익 부업 문의 텔레 주세요`
- 기대 결과: `critical`

### 정치글

- `요즘 정치 때문에 나라 망한다`
- 기대 결과: `medium` 이상

### 강한 비하/조롱 글

- 특정 집단/유저를 강하게 비하하는 글
- 기대 결과: `high` 이상

## 14. 1.5차 옵션: 사용자 라벨 버튼 임시안

DB 없이 빠르게 검증만 하고 싶다면 브라우저 `localStorage` 기반으로 임시 라벨링 UI를 붙일 수 있다.

방법:

- 상세 화면에 `분탕글`, `정상글` 버튼 표시
- `{ [postNo]: "troll" | "normal" }` 형태로 `localStorage` 저장
- 목록에 “내가 분탕 지정함” 표시 가능

제약:

- 브라우저별로 분리됨
- 서버 공유 불가
- 학습 데이터로 직접 활용하기 어려움

따라서 실제 운영 기능이라기보다 규칙 검증용 임시 장치로만 적합하다.

## 15. 최종 권고

1차 MVP는 아래 범위로 고정하는 것이 가장 좋다.

1. 규칙 기반 분석 모듈 추가
2. 목록에 간단한 분탕 위험도 배지 표시
3. 상세에 점수/등급/근거 표시
4. 오탐 방지용 감점 규칙 최소 포함

이 범위면 현재 구조를 크게 흔들지 않고 빠르게 도입할 수 있다. 사용자 라벨 저장과 학습은 다음 단계로 분리하는 편이 구현 난이도, 일정, 안정성 면에서 가장 합리적이다.
