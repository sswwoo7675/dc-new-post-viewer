# 분탕글 판별 기능 검토 문서

## 1. 목적

정보처리기사 미니 갤러리 새 글 뷰어에 다음 기능을 추가할 수 있는지 검토한다.

- 새로 올라온 글이 분탕글인지 아닌지 확률(%)로 표시
- 사용자가 글을 분탕글/정상글로 분류하는 버튼 제공
- 누적된 사용자 분류 데이터를 바탕으로 판별 성능 개선

이번 문서는 구현이 아니라 기술 검토와 개발 방향 제안이 목적이다.

## 2. 현재 코드베이스 기준 현황

현재 구조는 매우 단순하다.

- 서버는 Node.js 단일 파일 기반이며 DCInside 글 목록/본문/댓글을 수집해서 JSON API로 내려준다.
- 글 목록 API는 [server.js](/C:/project/dactest/server.js:413) 기준 `getPosts()`와 `/api/posts` 처리부에서 제공된다.
- 개별 글 API는 `parseArticle()`와 `/api/posts/:no` 처리부에서 제공된다.
- 개별 글 API의 본문/댓글 가공은 [server.js](/C:/project/dactest/server.js:296)와 [server.js](/C:/project/dactest/server.js:452) 부근에서 이뤄진다.
- 프론트는 [public/app.js](/C:/project/dactest/public/app.js:53)에서 목록을 렌더링하고, [public/app.js](/C:/project/dactest/public/app.js:81)에서 본문을 렌더링하는 구조다.
- 현재 영속 저장소(DB)가 없고, 브라우저 `localStorage`에는 읽음 여부만 저장한다.

현재 코드상 분탕글 판별 기능을 붙일 위치는 분명하다.

- 서버 측: 글 제목/본문/댓글을 기준으로 분류 점수를 계산해서 API에 포함
- 프론트 측: 목록과 상세 화면에 분탕 확률 및 사용자 분류 버튼 노출

하지만 아래 기능은 현재 구조에 없다.

- 분류 결과 저장
- 사용자 피드백 저장
- 모델 학습용 데이터셋 관리
- 재학습 실행 파이프라인

즉, `판별 결과를 보여주는 기능`은 비교적 빨리 가능하지만, `사용자 피드백을 학습으로 연결하는 기능`은 저장소와 학습 파이프라인이 먼저 필요하다.

## 3. 결론 요약

결론부터 말하면 기능 추가는 가능하다. 다만 난이도와 위험도가 서로 다르다.

### 바로 가능한 범위

- 규칙 기반(rule-based) 분탕글 점수 산정
- 제목/본문/댓글을 기준으로 0~100% 점수 표시
- 사용자에게 분탕글/정상글 분류 버튼 제공
- 사용자 분류 결과를 DB에 저장

### 추가 설계가 필요한 범위

- 누적 라벨을 이용한 자동 성능 개선
- 규칙 기반과 통계/ML 기반 모델의 혼합
- 재학습, 버전 관리, 오분류 보정

### 주의할 점

- “분탕글 확률 %”는 수학적으로 정확한 확률이라기보다 `모델 점수`에 가깝다.
- 초기에 표시되는 %는 실제 확률이라기보다 `분탕 가능성 점수`로 설명하는 편이 안전하다.
- “프로그램이 스스로 학습”은 실시간 자동 온라인 학습보다는, 일정량의 사용자 라벨이 쌓인 뒤 주기적으로 재학습하는 배치 방식이 더 현실적이고 안정적이다.

## 4. 요구사항별 가능 여부

### 4.1 분탕글 여부를 %로 표시

가능하다.

구현 방법은 크게 두 가지다.

1. 규칙 기반 점수화
- 금칙어, 광고 패턴, 정치 키워드, 특정 주제 폄하 표현, 과도한 비하/조롱 표현을 점수화
- 예: 광고 링크 포함, 반복 특수문자, 특정 문구 조합, 외부 홍보 패턴
- 장점: 빠르고 설명 가능성이 높다
- 단점: 우회 표현에 약하고 유지보수가 필요하다

2. ML/AI 기반 분류
- 제목/본문/댓글을 입력으로 받아 분탕글 확률을 예측
- 장점: 문맥 반영 가능
- 단점: 라벨 데이터가 필요하고 운영 복잡도가 올라간다

현 시점 추천은 `규칙 기반 + 간단한 점수 보정`으로 시작하고, 이후 사용자 라벨이 쌓이면 ML을 추가하는 방식이다.

### 4.2 예시 기준 필터링

가능하다. 다만 항목별 구현 난이도가 다르다.

#### 광고글

가장 구현이 쉽다.

주요 신호 예시:

- 반복 링크, 오픈채팅, 텔레그램, 디스코드 유도
- 특정 상업 키워드 반복
- 연락 유도 표현
- 짧은 본문 + 홍보 링크 조합
- 여러 글에서 유사 문구 반복

광고글은 규칙 기반만으로도 초반 성능이 비교적 잘 나오는 편이다.

#### “정보처리기사 필요없다/쓸모없다” 류 글

가능하지만 문맥 처리가 중요하다.

예를 들어 아래는 구분이 필요하다.

- 분탕: “정처기 왜 따냐 병신 자격증인데”
- 일반 토론: “실무에서는 정처기 효용이 낮다고 느꼈습니다”
- 정보글: “채용시장 기준으로 정처기 필요 여부를 비교해봤습니다”

따라서 단순 키워드 매칭만으로 처리하면 오탐이 많아질 수 있다. 이 항목은 다음이 필요하다.

- 키워드뿐 아니라 문장 톤 분석
- 비하 표현 강도 반영
- 제목만이 아니라 본문 전체 확인

초기에는 `특정 주제 폄하 + 공격적 표현 동반`일 때 점수를 높이는 방식이 적절하다.

#### 정치글

가능하다.

정치 관련 키워드, 정치인명, 정당명, 시사 이슈 키워드 등을 사전으로 관리하면 된다. 다만 일반 뉴스 링크나 비유 표현에서 오탐 가능성이 있다.

이 항목은 키워드 사전 관리만으로도 어느 정도 성능이 나오지만, 시의성 있는 키워드 추가 관리가 필요하다.

#### 매우 심한 비하 및 조롱

가능하지만 가장 어렵다.

이유는 다음과 같다.

- 디시 특성상 약한 조롱/비하는 허용 대상
- 따라서 일반 욕설 탐지와 분탕 탐지가 동일하지 않다
- “심한 수준”의 기준을 정량화해야 한다

이 항목은 아래처럼 다단계 기준이 필요하다.

- 욕설 존재 여부
- 특정 대상에 대한 반복 공격 여부
- 인신공격 강도
- 혐오/비하 표현의 조합 강도
- 본문 전체 분위기

즉, 단순 금칙어만으로는 부족하고 `강도 점수` 개념이 필요하다.

## 5. 추천 판별 방식

## 5.1 1단계: 규칙 기반 분류기

가장 먼저 도입하기 좋은 방식이다.

입력 데이터:

- 제목
- 본문 텍스트
- 댓글 수, 조회 수 같은 메타 정보
- 가능하다면 작성자 반복 패턴

점수 예시:

- 광고 패턴 매칭: +40
- 정치 키워드 다수 포함: +30
- 특정 자격증 폄하 문구 + 비하 표현 동반: +35
- 강한 비하/조롱 표현 다수: +20~50
- 반복 특수문자/도배 패턴: +10

최종 점수는 0~100으로 클램프해서 표시할 수 있다.

예시 출력:

- 0~29: 낮음
- 30~59: 주의
- 60~79: 높음
- 80~100: 매우 높음

이 단계의 장점은 설명 가능성이다.

- “정치 키워드 2개 + 강한 비하 표현 1개로 인해 72%”

이런 식으로 근거를 보여줄 수 있어 운영자가 기준을 수정하기 쉽다.

## 5.2 2단계: 사용자 피드백 수집

추천 버튼 구성:

- `분탕글`
- `정상글`
- 선택 취소 또는 `보류`

저장해야 할 데이터:

- 게시글 번호
- 제목
- 본문 텍스트
- 댓글 텍스트 요약 또는 전체
- 당시 계산된 규칙 점수
- 사용자가 누른 라벨
- 라벨링 시각

중요한 점은 `학습용 원문`을 함께 저장해야 한다는 것이다. 현재는 서버가 실시간으로만 글을 가져오므로, 나중에 재학습하려면 당시 본문을 다시 확보하지 못할 수 있다.

즉, 사용자 라벨을 받을 때는 최소한 아래 둘 중 하나가 필요하다.

- 글 원문 스냅샷 저장
- 학습에 필요한 텍스트 피처 저장

## 5.3 3단계: ML 분류기 추가

사용자 라벨이 일정량 이상 쌓이면 가능하다.

추천 접근:

1. 경량 텍스트 분류
- TF-IDF + Logistic Regression / Linear SVM
- 장점: 빠르고 데이터 적어도 시작 가능
- 한국어에서도 베이스라인 용도로 유효

2. 한국어 임베딩 기반 분류
- KoBERT, KR-SBERT, bge-m3 계열 임베딩 + 분류기
- 장점: 문맥 반영이 더 좋음
- 단점: 운영과 배포가 무거워짐

3. 외부 LLM/분류 API 활용
- 장점: 초기 정확도 확보가 쉬울 수 있음
- 단점: 비용, 지연시간, 외부 의존성, 정책 이슈

이 프로젝트 규모에서는 처음부터 대형 모델을 붙이기보다 아래 순서를 권장한다.

- 규칙 기반
- 라벨 수집
- 경량 ML
- 필요 시 임베딩/LLM 보강

## 6. “스스로 학습” 요구사항 해석

이 요구사항은 그대로 구현하면 오해의 소지가 있다.

### 비추천: 실시간 자동 온라인 학습

- 사용자가 한 번 라벨링할 때마다 즉시 모델을 다시 학습
- 오분류 라벨이 들어오면 성능이 쉽게 흔들림
- 운영 안정성이 낮음

### 추천: 주기적 배치 재학습

- 예: 라벨 100개 또는 300개가 쌓이면 재학습
- 검증 세트를 두고 성능이 좋아질 때만 새 모델 반영
- 모델 버전 기록

이 방식이 “스스로 학습” 요구사항을 가장 현실적으로 만족한다.

실무적으로는 아래가 필요하다.

- 라벨 데이터셋 저장
- 학습 스크립트
- 검증 지표(precision, recall, F1)
- 모델 파일 저장 및 버전 관리
- 운영 반영 승인 절차

## 7. 현재 구조에서 추가로 필요한 기술

## 7.1 저장소

현재는 서버 메모리 캐시와 브라우저 `localStorage`만 있다. 학습 기능을 위해서는 서버 측 영속 저장소가 필수다.

후보:

- SQLite
  - 가장 추천
  - 단일 서버/개인용 도구에 적합
  - 구현 난이도 낮음
- PostgreSQL
  - 다중 사용자/확장성 고려 시 적합
  - 현재 규모에는 다소 무거움

초기 추천은 SQLite다.

예상 테이블:

- `posts`
  - post_no, title, content_text, content_html, author, created_at, fetched_at
- `post_comments`
  - post_no, comment_id, author, text, created_at
- `troll_predictions`
  - post_no, score, label, reasons_json, model_version, created_at
- `post_labels`
  - post_no, user_label, source, created_at
- `model_versions`
  - version, type, metrics_json, created_at, is_active

## 7.2 텍스트 전처리

현재 서버는 HTML 파싱은 하지만, 분류용 텍스트 전처리 계층은 없다.

필요 기능:

- 제목/본문 HTML 제거 후 텍스트화
- 특수문자 정리
- URL, 연락처, 반복문자 패턴 추출
- 광고성 키워드 사전 매칭
- 정치 키워드 사전 매칭
- 비하/혐오/조롱 강도 사전 매칭

한국어 형태소 분석까지는 초기에 필수는 아니지만, 2단계 이후 성능 향상에는 도움이 된다.

후보 기술:

- 단순 정규식/사전 매칭
- `wink-nlp`, `natural` 같은 JS 라이브러리
- 별도 Python 학습 스크립트에서 `scikit-learn`, `konlpy`, `sentence-transformers`

## 7.3 분류 엔진

서버가 Node.js라서 선택지는 두 가지다.

1. Node.js 내부에서 규칙 기반 점수 계산
- 가장 간단
- 1단계에 적합

2. 학습/추론은 별도 Python 서비스 또는 배치 스크립트
- ML 도입 시 더 유리
- Node 서버는 결과 조회만 담당

추천 구조:

- 운영 API는 Node.js 유지
- ML 학습은 Python 스크립트 별도 운영
- 추론은 초기엔 Node 규칙 기반
- 추후 필요 시 Python 모델 결과를 가져오거나 ONNX 등으로 추론 통합

## 7.4 UI/UX

프론트에는 아래 추가가 필요하다.

- 목록에서 분탕 확률 배지 표시
- 상세 화면에서 점수와 판별 근거 표시
- `분탕글`, `정상글` 버튼
- 사용자가 이미 라벨링한 글인지 표시

특히 근거 노출은 중요하다. 오탐이 나왔을 때 왜 그렇게 판단했는지 보여주지 않으면 운영자가 기준을 수정하기 어렵다.

## 8. 추천 아키텍처

### A안. 규칙 기반 only

구성:

- Node 서버 내부 규칙 엔진
- SQLite에 판별 결과와 사용자 라벨 저장

장점:

- 빠르게 개발 가능
- 설명 가능성 높음
- 운영 단순

단점:

- 규칙 유지보수 비용이 커짐
- 우회 표현과 문맥 처리 한계

### B안. 규칙 기반 + 배치 학습형 ML

구성:

- 실시간: Node 규칙 엔진
- 저장: SQLite
- 학습: Python 배치 스크립트
- 배포: 학습된 모델 버전 관리

장점:

- 현실적인 정확도 개선 경로
- 초기 구현과 장기 확장성의 균형이 좋음

단점:

- 기술 스택이 2개(Node/Python)로 늘어남

### C안. 외부 LLM/API 기반 분류

구성:

- 새 글마다 외부 API 호출

장점:

- 빠르게 고도화된 판단 가능

단점:

- 비용
- 속도
- 외부 의존성
- 호출 제한
- 프라이버시/정책 검토 필요

현재 프로젝트에는 B안이 가장 적절하다.

## 9. 개발 단계 제안

### 1차

- 성능 지표 기반 교체
- 필요 시 임베딩/LLM 기반 고도화

## 10. 주요 리스크

### 10.1 분탕글 정의의 모호성

가장 큰 리스크다.

특히 아래는 합의가 필요하다.

- “정처기 무용론” 중 어디까지 분탕인가
- 정치 비유나 잡담은 어디까지 허용인가
- 디시 특유의 약한 조롱과 심한 조롱의 경계는 어디인가

라벨 기준이 불명확하면 학습 데이터 품질이 흔들린다.

### 10.2 라벨 품질

사용자 라벨이 적거나 일관되지 않으면 모델 성능이 오히려 나빠질 수 있다.

대응:

- 라벨 기준 문서화
- 취소/수정 기능 제공
- 초기에는 단일 운영자 라벨 위주로 운영

### 10.3 확률 수치의 신뢰성

사용자는 `%`를 실제 확률로 받아들이기 쉽다.

대응:

- 초기 UI 문구를 `분탕 가능성 72%`로 표시
- 상세에서 `자동 판별 점수`라고 설명

### 10.4 데이터 부족

초기에는 라벨 데이터가 적어서 ML보다 규칙 기반이 더 안정적일 가능성이 높다.

## 11. 권장 기술 스택

최소 권장안:

- 서버: 기존 Node.js 유지
- 저장소: SQLite
- 규칙 엔진: 순수 JavaScript
- 학습: Python + scikit-learn
- 모델 평가: precision / recall / F1

확장 권장안:

- 임베딩 기반 모델: sentence-transformers 계열
- 모델 서빙: Python API 또는 배치 결과 파일

## 12. 최종 권고

이 기능은 충분히 구현 가능하다. 다만 한 번에 “학습형 AI 분탕 판별기”로 가기보다 아래 순서가 가장 안전하다.

1. 규칙 기반 판별과 점수 표시부터 도입
2. 사용자 라벨 저장 구조를 만든다
3. 라벨이 쌓이면 배치 재학습형 ML을 붙인다
4. 정확도와 오탐률을 보면서 규칙 + ML 혼합 방식으로 발전시킨다

현재 코드베이스 기준으로는 1차와 2차까지는 비교적 자연스럽게 붙일 수 있다. 반면 “스스로 학습”은 저장소, 라벨 품질 관리, 재학습 파이프라인까지 포함해야 하므로 별도 단계로 보는 것이 맞다.

즉, 이 기능은 **가능하다**. 다만 추천 개발 방향은 **규칙 기반 MVP -> 사용자 라벨 수집 -> 배치 학습형 고도화** 순서다.
